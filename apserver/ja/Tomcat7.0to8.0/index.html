1.はじめに


1.1.本ドキュメントについて


本マイグレーションガイドはTomcat間（7.0.x -> 8.0.x）のマイグレーション時に注意すべき項目について整理したものである。

本内容はTomcatコミュニティのMigration Guideの情報を参考に作成している。

http://tomcat.apache.org/migration.html

http://tomcat.apache.org/migration-8.html

コミュニティのMigration Guide はマイナーバージョンアップ時やバグ修正時に適宜更新されているため、最新情報はコミュニティページを参照のこと。



1.1.1.Javaバージョンについて


Apache Tomcatのあるメジャーバージョンから新しいバージョンのものへアップデートをするとき、システムにインストールされているJVMが必要なJavaバージョンをサポートしていることを確認する必要がある。

サポートするJavaバージョンは以下で確認できる。

http://tomcat.apache.org/whichversion.html



1.1.2.設定ファイルについて


Apache Tomcatのあるメジャーバージョンから他のバージョンへマイグレーション（例：Tomcat 7からTomcat 8）するとき、古いバージョンの設定ファイルをむやみにコピーすべきではありません。

推奨されているアプローチはApache Tomcatの新しいバージョンのデフォルト設定ファイルを必要に応じて修正する方法である。

本ドキュメントでは対象外であるが、同じメジャーバージョン内であるマイナーバージョンから別のマイナーバージョンへマイグレーション（例；Tomcat 7.0.14からTomcat 7.0.16）するとき、設定ファイルは再利用することが可能である。

しかし、デフォルト値が変更されていないか、また、新たな要素が追加されていないかを確認するべきである。

確認結果に応じて設定ファイルを修正するべきである。



2.Java7


2.1.Java7必須


Tomcat8.0は、Java7以降が必須となる。

古いバージョンのJavaを利用している場合はアップデートを行う必要がある。

Tomcat8.0は、Servlet 3.1、JSP2.3、EL3.0およびWebSocket1.1仕様をサポートしている。

http://tomcat.apache.org/whichversion.html

これらの仕様で非推奨となったAPIを利用している場合は代替APIを利用するように変更する必要がある。

3.1.2.JSP非推奨API


チェック対象は非推奨となったAPIである。以下のドキュメントを参照して、代替APIを利用するように変更する。

http://docs.oracle.com/javaee/7/api/deprecated-list.html



3.1.3.EL非推奨API

3.1.4.Servlet、JSP以外の非推奨API

http://docs.oracle.com/javaee/7/api/deprecated-list.html



3.2.Servlet 3.1 API


JSPページにおいて、import属性（<%@page import="パッケージ名" %>）のパッケージ名にワイルドカード（*）を使用している場合、Servlet APIに新しく追加されたクラスとWebアプリケーション内のクラスが同名クラスであると競合する可能性がある。

例えば、パッケージ"tubame.example"にReadListenerクラスが含まれている場合、以下のJSPページはTomcat8でコンパイルできなくなる。

<%@page import="tubame.example.*"%>
<% ReadListener listener = new ReadListener(); %>

Tomcatでは、JSPからServlet（Java）にコンパイルする際に、「javax.servlet.*」「javax.servlet.http.*」「javax.servlet.jsp.*」のインポートを設定するため、Servlet3.1で当該パッケージに新たに追加された以下のクラスと、同名クラスを持つパッケージをワイルドカードを使用してインポートした場合に衝突してしまう。

  • ReadListener

  • WriteListener

  • HttpSessionIdListener

  • HttpUpgradeHandler

  • WebConnection

本問題の対処方法は、明示的なインポートにおいて、クラス名まで指定することである（上記例では「import="tubame.example.ReadListener"」とする）。



3.3.JavaServer Pages 2.3


統一された式言語（Java Unified Expression Language 3.0）はstatic フィールドおよびstatic メソッドを参照するためのサポートが追加されました。

JSP で本機能をサポートするためには、インポートされたクラスやフィールドの名前が存在するかどうか確認するためにID をチェックするようにjavax.servlet.jsp.el.ScopedAttributeELResolver 実装の変更が必要でした。

いくつかの状況において、本変更は重大なスローダウンを引き起します。

これは、ページスコープ、リクエストスコープ、セッションスコープ、もしくはアプリケーションスコープの変数を参照する、もしくはスコープ未定義のID に影響を与えます。

未定義の場合、インポートされたクラスもしくはフィールドが存在するかどうかを確認するためにチェックされるので、ID を解決するために大幅に時間がかかります。

以上のスローダウンを回避するために、次のコード：

${undefined}

は次のコードに置換える必要があります。

${requestScope.undefined}

もしくは、変数が定義されているスコープに対して適切にスコープを使用する必要があります。



4.設定項目の変更


4.1.デフォルトコネクタ


デフォルトのHTTP/AJPコネクタ実装はBIO (Java blocking IO implementation) コネクタから NIO (Java non-blocking IO implementation) コネクタに変更されている。

BIOコネクタとNIOコネクタの違いについては、次のURLを参照すること。

http://tomcat.apache.org/tomcat-8.0-doc/config/http.html

http://tomcat.apache.org/tomcat-8.0-doc/config/ajp.html



4.2.Web Application Resources


古いバージョンで使用されていたAliases、VirtualLoader、VirtualDirContext、外部Repositories、JAR resources などの機能は単一のResourcesとしてリファクタリングされた。



4.2.1.Context要素の属性


Context要素で指定していた以下の属性はResources要素で指定する。

  • allowLinking

  • cachingAllowd

  • cacheMaxSize

  • cacheObjectMaxSize

  • cacheTTL

例として、Tomcat7とTomcat8の設定例を以下に示す。

<!-- Tomcat 7: -->
<Context allowLinking="true" />

<!-- Tomcat 8: -->
<Context>
<Resources allowLinking="true" />
</Context>


4.2.2.Resoucesの指定方法変更


Aliases、VirtualLoader、VirtualDirContext、External repositories、JAR resourcesはResources要素の子要素（PreResources、PostResources、JarResources）で指定する。

新たな実装の使用方法については以下のドキュメントで提供されている。

http://tomcat.apache.org/tomcat-8.0-doc/config/resources.html



4.3.DBCPバージョンアップ


Tomcat同梱版のDBコネクションプールがDBCP1.4からDBCP2に変更された。



4.3.1.JDBCドライバのバージョン


DBCP2.0 はJDBC4.1 をサポートしている。

DBCP2.0 では互換性を持っているため、JDBCの下位バージョンのドライバでも問題なく動作するはずではあるが、一部JDBC4.1 用のインタフェースを持っているため、JDBCドライバのアップデートを検討すること。



4.3.2.設定項目の変更


DBCP1.4で使用されていたいくつかのパラメータ名が、DBCP2で変更されている。

  • maxActive（コネクションの最大生成数）はmaxTotalに変更された

  • maxWait（コネクション取得時の最大待ち時間(ミリ秒)）はmaxWaitMillisに変更された

  • removeAbandonedはremoveAbandonedOnBorrow（コネクションを取得するときにremoveAbandoned()を実施するかどうかを設定）およびremoveAbandonedOnMaintenance（EvictorスレッドによってremoveAbandoned()を実施するかどうかを設定）に変更



4.3.3.JDBCドライバ配置場所


JDBCドライバが特定のWeb アプリケーションでのみ使用される場合は、$CATALINA_BASE/lib ではなく、WEB-INF/lib に配置する。



4.3.4.Validation方式の変更


以前のバージョンではvalidationQueryとtestXxxの両方が設定されていないとConnectionの検証が行われなかった。

DBCP2.0からはいずれかのtestXxxがtrueになっていれば検証が行われる。この時、validationQueryが定義されていない場合は、Connection.isValid()を使用して検証が行われる。



4.4.クラスタリング



4.4.1.セッションID変更


Servlet 3.1でのHttpServletRequest.changeSessionId()メソッドの追加により、org.apache.catalina.ha.session.JvmRouteSessionIDBinderListenerは不要になり、削除された。

Tomcat8へのアップグレード時にはクラスタ設定からJvmRouteSessionIDBinderListenerを削除する必要がある。



4.4.2.SessionIdGeneratorの追加


セッションIDの生成を拡張可能にするための新たなインタフェースSessionIdGeneratorが追加された。

SessionIdGeneratorの追加に伴い、ManagerのsessionIdLength属性とManager.getSessionIdLength()およびManager.setSessionIdLength(int)は非推奨となった。

セッションID長のコンフィグレーションを使用している場合はSessionIdGenerator要素のsessionIdLength属性を使用する。



4.5.CredentialHandlerの追加


提供されたパスワードとRealmが格納したパスワードを比較するための新たなインタフェースCredentialHandler が追加された。

CredentialHandler の追加に伴い、Realm のdigest 属性は非推奨になった。

ダイジェスト化されたパスワードを使用している場合は、<Realm>の子要素に<CredentialHandler> を定義する必要がある。



5.その他


5.1.Debug用ポート


リモートデバッグを有効にするためにjpda オプションを指定して起動した場合、従来は*:8000で待ち受けていたが、localhost:8000に変更となった。

値を変更したい場合は、setenvスクリプトなどを使用して環境変数JPDA_ADDRESSに設定することで上書き可能である。



5.2.内部APIの変更


内部APIが以下のように変更された。

  • Manager、Loader、ResourcesがContainerからContextに移動

  • MapperがConnectorからServiceに移動

  • 各機能を分離するのではなく、単一のフレームワークとしてAliases、VirtualLoader、Virtual、DirContext、JAR resourcesおよびexternal repositoriesをマージするためのResourcesが新たに実装



5.3.Deployment


WebアプリケーションがWARファイルとしてデプロイされており、WARファイルを展開するように設定されていない場合、Tomcat 起動時に大幅に遅延（3～10倍）が発生し、実行パフォーマンスの大幅な遅延（ユーザアプリケーションの構造に依存）を引き起こします。

Host 要素の属性unpackWARs にfalse、もしくはContext 要素の属性unpackWAR にfalse を設定しないことを推奨します。
